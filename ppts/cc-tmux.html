Chapter One
为什么需要 tmux？

单一终端窗口限制了 Claude Code 的真实能力——tmux 将改变这一切。

Chapter 01 · 背景与动机

tmux 解决了什么问题

Claude Code 是长期运行的 AI Agent，而不是单次命令工具。

🔌
会话持久化
SSH 断线、网络抖动不再中断长时间任务。Claude Code 工作进程后台安全运行，随时重新附加。
⚡
并行工作流
左侧窗格 Claude 生成代码，右侧实时运行测试，底部观察日志——三件事同时进行。
🔁
上下文隔离
为不同功能、分支分配独立窗口，保持每个 Claude 会话的上下文干净聚焦。
🤖
脚本自动化
通过 tmux send-keys 向 Claude Code 会话发送命令，实现 CI/CD 集成和批量任务编排。
核心理念
tmux 正是为长期运行的进程而生。两者的结合不是可选项，是最佳实践。

Chapter 02 · 安装与配置

推荐的 ~/.tmux.conf

专为 Claude Code 优化的配置，提升日常使用体验。

~/.tmux.conf · Claude Code 优化版
# 前缀键改为 Ctrl+a（更符合人体工学）
set -g prefix C-a  unbind C-b  bind C-a send-prefix

# 鼠标支持 + 256色真彩色
set -g mouse on
set -g default-terminal "tmux-256color"
set -ag terminal-overrides ",xterm-256color:RGB"

# 大历史缓冲区（Claude 输出很长！）
set -g history-limit 50000

# 直觉化分割键
bind | split-window -h -c "#{pane_current_path}"
bind - split-window -v -c "#{pane_current_path}"

# Vim 风格导航
bind h select-pane -L  bind l select-pane -R
bind j select-pane -D  bind k select-pane -U

# 状态栏 + 重载
set -g status-style 'bg=#1a1e28,fg=#8892a4'
set -g window-status-current-style 'bg=#00d4aa,fg=#000,bold'
bind r source-file ~/.tmux.conf \; display "✓ 配置已重载"

Chapter Three
tmux 三层结构

理解 Session → Window → Pane 的层次关系，是高效使用的基础。

Chapter 03 · 核心概念

Session / Window / Pane

每个 Git 功能分支对应一个 tmux Window，是最推荐的组织方式。

Session: claude-project
├── Window 0: feature/auth
│   ├── Pane 0: claude   ← AI 对话
│   └── Pane 1: pytest --watch
├── Window 1: feature/api
│   ├── Pane 0: claude
│   ├── Pane 1: uvicorn --reload
│   └── Pane 2: httpie / curl
└── Window 2: monitoring
    ├── Pane 0: htop
    └── Pane 1: tail -f logs/app.log
💡 最佳实践
用 Prefix + , 将 Window 重命名为分支名，多任务切换时不迷失方向。

Chapter 04 · 快捷键速查

Session & Window 操作

Session

快捷键功能
Prefix + d分离（后台保持运行）
Prefix + s列出所有会话并切换
Prefix + $重命名当前会话
tmux ls查看所有会话
tmux attach -t 名重新附加到会话

Window

快捷键功能
Prefix + c新建窗口
Prefix + ,重命名当前窗口
Prefix + n / p下 / 上一个窗口
Prefix + 0-9按编号跳转
Prefix + w窗口列表预览
使用场景
每次开始新功能：Prefix+c 新建窗口 → Prefix+, 命名为分支名 → 启动 Claude

Chapter Five
黄金工作流模式

三种经过验证的模式，适用于不同规模和类型的开发任务。

Chapter 05 · Pattern A

🔥 代码生成 + 测试反馈循环

最常用模式 · 适合日常功能开发

一键初始化标准开发会话
# 创建会话并分割三窗格
$ tmux new-session -d -s dev -n claude
$ tmux split-window -h -t dev:claude   # 右侧窗格
$ tmux split-window -v -t dev:claude.1 # 右下窗格

# 各窗格启动对应进程
$ tmux send-keys -t dev:claude.0 'claude' Enter
$ tmux send-keys -t dev:claude.1 'npm test -- --watch' Enter
$ tmux send-keys -t dev:claude.2 'git log --oneline -10' Enter

# 附加并开始工作
$ tmux attach-session -t dev
工作循环
Claude 生成代码 → 测试窗格自动检测变更 → 看到失败 → 切换 Claude 窗格粘贴错误 → Claude 修复 → 重复
Chapter 05 · Pattern B

🚀 多 Agent 并行开发

高级模式 · 多功能并行，显著提升整体速度

多分支并行会话
# 为每个功能分支创建独立 Claude 会话
$ tmux new-session -d -s auth
$ tmux send-keys -t auth 'git checkout feature/auth && claude' Enter

$ tmux new-session -d -s api
$ tmux send-keys -t api 'git checkout feature/api && claude' Enter

$ tmux new-session -d -s ui
$ tmux send-keys -t ui 'git checkout feature/ui && claude' Enter

# 查看所有并行会话 · 快速切换
$ tmux ls                         auth: 1w · api: 1w · ui: 1w
$ tmux switch-client -t api
⚠️ 注意
并行多个 Claude 实例消耗更多 API 用量，建议先掌握单 Agent 流程再扩展。
Chapter 05 · Pattern C

🤖 自动化脚本驱动

CI/CD 集成 · 无人值守批处理

auto-claude.sh
#!/bin/bash
SESSION="claude-auto"  TASK="$1"

if ! tmux has-session -t $SESSION 2>/dev/null; then
  tmux new-session -d -s $SESSION
  tmux send-keys -t $SESSION 'claude' Enter
  sleep 3  # 等待 Claude 初始化
fi

# 发送任务并附加观察
tmux send-keys -t $SESSION "$TASK" Enter
tmux attach -t $SESSION

# 使用: ./auto-claude.sh "审查 src/ 目录的安全漏洞"

Chapter Six
Claude Code 深度技巧

充分利用 Claude Code 的高级特性，在 tmux 环境中发挥最大效能。

Chapter 06 · 深度技巧

五大高效组合技

1
会话恢复后的上下文重建
SSH 断线后 tmux attach -t dev 恢复，Claude 对话历史完整保留。使用 /resume 要求 Claude 总结进度并继续。
2
跨窗格的错误注入
测试窗格失败后，Prefix + [ 复制模式选中错误信息，Prefix + ] 粘贴到 Claude 窗格，即时修复。
3
CLAUDE.md 统一多 Agent 规范
根目录的 CLAUDE.md 定义架构约束、代码风格。所有并行 Claude 实例自动读取，确保多 Agent 输出一致性。
4
--print 模式管道集成
非交互窗格使用 claude -p "分析代码" < file.py | tee report.md 将 AI 分析输出到文件，供其他工具消费。
5
send-keys 实现远程 Claude 编排
通过 tmux send-keys -t session "任务描述" Enter 从脚本或 CI 系统远程驱动 Claude 完成任务。
Chapter 06 · 深度技巧

CLAUDE.md 项目规范

放置于项目根目录，所有 Claude 实例自动遵守。

CLAUDE.md · 示例
## 技术栈
- Python 3.11, FastAPI, PostgreSQL, Redis
- 测试: pytest + httpx    代码风格: Black + ruff

## 架构规范
- 数据库操作必须通过 Repository 模式
- 业务逻辑放在 services/ 层，禁止出现在路由中
- 敏感配置从环境变量读取，禁止硬编码

## 禁止事项
- 不得使用同步 ORM  不得引入全局状态  不得跳过类型注解

## 测试要求
- 新功能必须包含单元测试，覆盖率 ≥ 80%
- API 端点必须有集成测试

Chapter 07 · 动手演练

三级难度练习

🟢
练习 1 · 基础（15 min）
创建名为 training 的 tmux 会话，分割 2 个窗格，左侧启动 Claude Code，右侧运行 watch -n1 date。练习窗格切换、最大化还原、重命名窗口。
🟡
练习 2 · 进阶（30 min）
用 Claude Code 创建简单 Todo API。三窗格：左侧 Claude 生成代码，右上 httpie 测试，右下监控日志。体验完整开发反馈循环。
🔴
练习 3 · 挑战（45 min）
编写 shell 脚本，一键创建 3 个功能分支的并行 Claude 会话，每个自动切换对应分支并启动 Claude。加分：断线后恢复所有会话。
⚠️ 注意事项
并行多个 Claude Code 实例会消耗更多 API 用量。学习阶段建议从单会话开始，熟悉后再扩展到多 Agent 模式。
