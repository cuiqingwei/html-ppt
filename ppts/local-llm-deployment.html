MODULE 01 · 01
什么是 LLM？
大语言模型（Large Language Model）的核心概念
🤖 传统 AI
规则驱动，针对特定任务训练
如：图像分类、语音识别、推荐系统

窄 AI 任务专用
✨ 大语言模型
海量文本训练，涌现通用理解与生成能力
写作、代码、问答、推理……一模多用

通用 AI 涌现能力
💡 核心原理
LLM 的本质是一个概率预测机器：给定前文，预测下一个最可能出现的 Token。海量参数+海量数据让它在这个看似简单的任务中"学会"了语言理解、逻辑推理。
📌 关键词
Transformer 架构 自注意力机制 预训练 + 微调 RLHF 对齐
MODULE 01 · 02
Token 与上下文窗口
理解 LLM 的"感知单元"与"记忆上限"
🧩 Token 是什么？
Token ≠ 字符，≠ 单词
中文：约 1.5~2 字 / Token
英文：约 0.75 词 / Token

"你好世界" → 大约 4 Tokens
"helloworld" → 大约 2 Tokens
⚠️ 计费与限速
API 调用按 Token 计费；本地部署的速度（tok/s）也以此为单位衡量
📐 上下文窗口（Context Window）
模型一次能"看到"的最大 Token 数

4K 适合简单问答
32K 可处理中等长度文档
128K+ 长文档、多轮对话
📌 本地部署影响
上下文越长，占用 VRAM 越多（KV Cache）。本地跑 8K 比 128K 省很多显存
MODULE 01 · 03
参数量与量化
7B / 13B / 70B 意味着什么？Q4 / Q8 / FP16 怎么选？
📊 参数量（B = Billion）
7B — 入门友好~4GB VRAM
13B — 性价比首选~8GB VRAM
34B — 高质量推理~20GB VRAM
70B — 接近 GPT-4~40GB VRAM
🗜️ 量化（Quantization）
FP32 原始精度，训练用，极耗显存
FP16 推理常用，质量高，显存 × ½
Q8 量化 8bit，几乎无损，显存 × ¼
Q4 量化 4bit，本地主流，显存 × ⅛
Q2 极度压缩，质量下降明显
✅ 推荐策略
日常使用 Q4_K_M 格式：质量与显存的最佳平衡点
格式每参数存储7B 显存需求质量损失适用场景
FP162 字节~14 GB无高端 GPU
Q8_01 字节~7 GB极少8GB 显卡
Q4_K_M0.5 字节~4.5 GB较少✅ 推荐日常
Q2_K0.25 字节~2.5 GB较大极限省显存
🔢 显存需求快速换算公式
📐 基础公式
模型权重显存 = 参数量(B) × 每参数字节数
FP16：× 2　Q8：× 1　Q4：× 0.5
KV Cache 显存 ≈ 上下文长度(K) × 0.5 MB
8K 上下文约需额外 ~4 GB
总显存需求 = 权重 + KV Cache + ~10% 余量
🧮 实际换算示例
Qwen2.5-7B Q4（4K上下文）
7 × 0.5 + 2 + 0.5 ≈ 6 GB ✅ 8GB 卡可用
DeepSeek-R1-14B Q4（8K上下文）
14 × 0.5 + 4 + 0.7 ≈ 11.7 GB ✅ 16GB 卡可用
Llama-3.1-70B Q4（4K上下文）
70 × 0.5 + 2 + 3.5 ≈ 40.5 GB ⚠️ 需双卡 3090

MODULE 03 · 01
GPU 选购核心指标
LLM 推理与游戏/图形渲染的需求截然不同
🥇 显存容量（VRAM）— 最重要
决定能跑多大的模型
模型权重 + KV Cache 都存在显存里
显存不够 = 模型跑不起来（或极慢）
🥈 显存带宽 — 决定速度
决定 Token 生成速度（tok/s）
LLM 推理是带宽密集型，非算力密集型
带宽越大 = 回复越流畅
GPUVRAM带宽可跑模型规模参考价格推荐度
RTX 40608 GB272 GB/s7B Q4~2500元入门
RTX 4060 Ti16 GB288 GB/s13B Q4~3800元推荐
RTX 309024 GB936 GB/s34B Q4~5000元高性价比
RTX 409024 GB1008 GB/s34B Q4~13000元发烧级
双卡 309048 GB1872 GB/s70B Q4~10000元进阶多卡
⚠️ 注意
AMD GPU 理论上可用（ROCm），但驱动生态远不如 NVIDIA CUDA 成熟，新手建议坚持 N 卡。

CPU 和内存也很重要——别让 GPU 闲着

  • CPU: i5-13600K / R5-7600
  • RAM: 32GB DDR5
  • SSD: 1TB NVMe
  • 可运行 Qwen2.5-7B Q4

⚖️ 主力方案 ~¥15,000

✅ 最推荐配置
  • GPU: RTX 3090 24GB
  • CPU: i7-13700K / R7-7700X
  • RAM: 64GB DDR5
  • SSD: 2TB NVMe
  • 可运行 34B Q4 / 13B FP16
性价比最优

🚀 进阶方案 ~¥25,000+

70B 级别模型
  • GPU: RTX 4090 × 1 或 3090 × 2
  • CPU: i9-13900K / TR 系列
  • RAM: 128GB DDR5
  • SSD: 4TB NVMe
  • 可运行 70B Q4 全量
📌 CPU + RAM 的作用
当模型太大无法完全放入显存时，llama.cpp 会将部分层卸载到 CPU/RAM（CPU Offload）。RAM 越大，能跑的模型越大，但速度会下降。64GB RAM 是舒适起点。

MODULE 04 · 03
MLX 生态 — Apple Silicon 的专属加速
mlx-lm 是基础，oMLX 是为 Agent 场景打造的生产级增强版
🧰 mlx-lm  Apple 官方
Apple 开源的 MLX 推理工具包，底层框架
支持 MLX / GGUF 量化格式
提供命令行推理 + 简单 HTTP 服务
速度约为 llama.cpp 的 1.5~2×
仅限 Mac M 系列
bash — mlx-lm
$ pip install mlx-lm $ mlx_lm.generate \ --model mlx-community/Qwen2.5-7B-Instruct-4bit \ --prompt "你好" $ mlx_lm.server \ --model mlx-community/Qwen2.5-14B-Instruct-4bit \ --port 8080 ✔ http://localhost:8080 OpenAI 兼容
🚀 oMLX  Agent 优化
基于 MLX 的 macOS 原生生产级推理服务器
核心亮点：SSD 分页 KV 缓存——把 KV Cache 持久化到磁盘，跨请求、跨重启可复用，Agent 多轮对话第二次响应 <5 秒（vs 普通方案 30~90 秒）
连续批处理，8 并发可达 4.14× 吞吐提升
支持 OpenAI + Anthropic 双协议兼容
原生菜单栏 App，已签名公证
bash — oMLX 从源码安装
# 方式一：下载 DMG 安装包（推荐） https://github.com/jundot/omlx/releases # 方式二：从源码安装 $ git clone https://github.com/jundot/omlx $ cd omlx && pip install -e . $ omlx serve --model-dir ~/models ✔ http://localhost:8000 OpenAI + Anthropic 兼容
⚡ oMLX vs mlx-lm 核心差异
mlx-lm 每次请求都从头计算 KV Cache；oMLX 将 Cache 块写入 SSD，再次遇到相同前缀直接从磁盘恢复——Coding Agent 反复读取同一代码库时效果最显著
📌 适用场景选择
单次问答 / 简单脚本 → mlx-lm
Claude Code / Cursor 本地后端 → oMLX
GUI 对话 → LM Studio / Ollama
需要 Anthropic API 格式 → oMLX（原生支持 /v1/messages）

📊 GPU 实时监控 — nvtop
🖥️ 什么是 nvtop？
类似 htop 的 GPU 实时监控工具
显示显存占用、GPU 利用率、温度、功耗
支持多卡并排显示，推理时必备
开源免费 Linux / Mac
bash — nvtop 安装与使用
# Ubuntu / Debian / macOS安装 $ sudo apt install nvtop 或 brew install nvtop # 启动监控 $ nvtop GPU 0 RTX 3090 [████████░░] 78% MEM 18432/24576 MiB ⚡ 320W 🌡 72°C
📌 推理时重点关注的指标
MEM Used：模型是否完全载入显存（越满越好，溢出则 CPU Offload 变慢） ·  GPU Util%：生成 Token 时应 >80%，若接近 0% 说明在等 CPU ·  Temp：长时推理注意散热，超过 85°C 考虑降频保护

MODULE 06 · 02
性能调优：让模型跑得更快
影响推理速度的关键参数与实战技巧
参数含义建议值影响
num_gpu / -ngl卸载到 GPU 的层数尽量最大↑↑↑ 速度
context_length上下文窗口大小4096~8192越长越慢
num_threadCPU 线程数（CPU Offload 时）物理核心数CPU 推理速度

常见问题与踩坑汇总
前人踩过的坑，让你少走弯路
🚨 CUDA out of memory
显存不足最常见报错。解决：①换更小量化版本（Q4 → Q2）②降低 context_length ③开启 CPU Offload（-ngl 减小层数）④关闭其他 GPU 应用
🚨 模型输出乱码 / 重复
通常是 ①Temperature 设置过高（调低到 0.6~0.8）②模型格式与框架不匹配 ③使用了错误的聊天模板（Chat Template）
🚨 速度极慢（<1 tok/s）
模型在 CPU 上运行而非 GPU。检查：nvidia-smi 推理时 GPU 利用率是否 >0，-ngl 参数是否正确设置
🚨 连接断开 / 超时
长上下文或大模型推理时间过长触发超时。增大客户端 timeout 设置；或缩短上下文 / 使用更小模型
📌 日常运维小技巧
用 ollama ps 查看正在运行的模型 · ollama stop <model> 释放显存 · 多个模型不要同时加载 · 推理完立刻关闭不用的服务
MODULE 06 · 04
微调入门：LoRA / QLoRA
在消费级 GPU 上定制专属模型
🔧 什么是微调？
在预训练模型基础上，用少量特定数据继续训练，让模型在特定领域表现更好。

全量微调需要与预训练相当的资源——消费级 GPU 不现实。
💡 LoRA / QLoRA 方案
LoRA：只训练少量"适配器"矩阵，参数量仅占全量的 0.1%~1%
QLoRA：基础模型量化为 4bit，再套 LoRA，显存需求大幅降低

24GB GPU 可微调 7B
微调工具特点适用场景
LLaMA-FactoryWeb UI + 命令行，中文支持好✅ 入门首选
Unsloth速度最快（2x），显存最省消费级 GPU
axolotl配置灵活，功能全面高级定制
Swift (ModelScope)阿里出品，Qwen 系列支持最好国内生态
📌 最小数据集建议
高质量指令数据 500~2000 条即可看到效果。数据质量 >> 数据数量。
